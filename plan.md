# Plan for CorePost

## Роль репозитория

Корневой репозиторий должен быть точкой входа в проект: описание системы, карта компонентов, диаграммы, документация по запуску демо и ссылки на релизы.

## Текущее состояние

- `README.md` описывает старую структуру (`initramfs`, `daemon`, один `mobile`) и не совпадает с актуальной моделью проекта.
- В документации нет актуального дерева из 7 репозиториев.
- Нет единой матрицы версий и ссылок на релизы Android/iOS.

## Что сделать

- Обновить `README.md` под актуальную модель: `CorePost`, `corepost-server`, `corepost-preboot`, `corepost-install`, `corepost-agent`, `corepost-mobile-android`, `corepost-mobile-ios`.
- Синхронизировать описание архитектуры, терминологию и сценарии с текущей структурой и целевыми сценариями проекта.
- Добавить карту репозиториев, зависимостей и ответственности каждого компонента.
- Описать единый demo flow: регистрация, штатная загрузка 2FA/3FA, panic-lock, post-boot отзыв доверия, восстановление.
- Добавить ссылки на артефакты релизов Android/iOS, инструкции по запуску стенда и требования к VM/QEMU-сценариям.
- Зафиксировать соглашения по именам репозиториев и веток, включая переименование `corepost-initramfs` -> `corepost-preboot` и приведение `corepost-server` к `main`.

## Критерий готовности

- Корневой репозиторий соответствует актуальной структуре и отражает все 7 модулей.
- Новый участник проекта может из одного `README.md` понять архитектуру, порядок запуска стенда и где искать релизные артефакты.
